## 階段一：關鍵缺陷修復（優先處理）

### 1.1 SVG `<defs>` ID 衝突

- **現狀**：所有敵人/塔共用 `nGrad`、`tGrad`、`bbGrad` 等固定 ID，多個 SVG 插入 DOM 時 `<linearGradient>` 互相覆蓋，導致圖形渲染錯誤
- **解法**：`generateSVG` 使用 `key` 參數動態產生唯一 ID（例如 `grad_${key}`），確保每個 SVG 實例的 `<defs>` 彼此獨立

### 1.2 `gameLoop` 取消機制

- **現狀**：`requestAnimationFrame` 在遊戲結束或切換模式時未呼叫 `cancelAnimationFrame`，可能導致多個動畫迴圈同時執行
- **解法**：儲存 `animFrameId`，在 `gameOver`、`gameWin`、`startGame` 開頭呼叫 `cancelAnimationFrame`，確保舊迴圈終止後才啟動新一局

### 1.3 `startGame` 重啟時全域陣列未清理

- **現狀**：`towers`、`enemies`、`particles` 等在 `startGame` 內有重置，但若前一個 `gameLoop` 尚未被取消而仍殘留，會繼續操作已被重置的陣列
- **解法**：配合 1.2 的取消機制，確保舊動畫迴圈完全終止後才建立新迴圈

## 階段二：效能與品質提升

### 2.1 DOM 參考快取

- **現狀**：反覆出現 `document.getElementById` 查詢，每個 frame 可能重複執行多次
- **受影響元素**：`shopScreen`、`homeScreen`、`gameUI`、`overlayScreen`、`nextWaveBtn`、`sellBtn`、`pauseBtn`、`muteBtn`、`scalingInfo`、`warpOverlay`、`mergeHint`、`modeDisplayText`、`moneyDisplay`、`livesDisplay`、`waveDisplay`、`bossKillDisplay`、`towerCountDisplay`、`towerMaxDisplay`、`bankDisplays`（home/shop）
- **解法**：初始化階段一次性快取所有 DOM 參考，後續直接使用變數

### 2.2 塔防行為策略表改為查表

- **現狀**：
  - `TowerGraphics.generateSVG` 含 26 條 `if-else` 判斷塔防武器 SVG
  - `Tower.getDamage()` 使用 `switch` 判斷各塔防等級傷害
- **解法**：改為設定物件查表

  ```javascript
  const TOWER_WEAPON_SVGS = {
      basic: (c) => `<rect ...>`,
      sniper: (c) => `<rect ...>`,
      // ...
  };

  const TOWER_DAMAGE_PER_LEVEL = {
      nuke: 100, cannon: 25, single: 30, flame: 1, // ...
  };
  ```

### 2.3 合併 `gameLoop` 中的雙重 `towers.forEach`

- **現狀**：每個 frame 對 `towers` 走訪兩次（先重置 buffs，再走訪一次計算 radar），可合併為單次走訪
- **解法**：在一次 `forEach` 內完成 buff 重置與 radar 計算

### 2.4 `Projectile.isInstant` 簡化

- **現狀**：以雙層條件判斷決定是否為即時攻擊（nuke / single），可改用 Set 查詢
- **解法**：`new Set(['nuke', 'single']).has(this.type)`

### 2.5 特效陣列清理優化

- **現狀**：`particles`、`effects`、`floatingTexts` 每個 frame 從尾部反向遍歷並使用 `splice` 刪除，大量粒子時開銷大
- **解法**：改用 `filter` 或雙指標壓縮，降低 splice 重複操作成本

## 階段三：結構與可維護性

### 3.1 抽離設定檔

- **現狀**：`TOWER_TYPES`、`ENEMY_TYPES`、`PATH_POINTS`、`UNLOCK_SCHEDULE` 與遊戲邏輯混在同一檔案中
- **解法**：將設定資料移至獨立區塊或獨立 JS 檔，與邏輯分離

### 3.2 遊戲狀態重設函數

- **現狀**：`startGame` 中 gameState 重建（1299-1302 行）使用直接物件字面量，容易遺漏欄位
- **解法**：提取 `createDefaultState(mode)` 函數，集中管理初始狀態結構

### 3.3 戰役模式解鎖邏輯釐清

- **現狀**：`checkUnlocks` 僅對無限模式有效，戰役模式依賴 `UNLOCK_SCHEDULE` + `handleWaveClear`，兩者行為不一致
- **解法**：統一解鎖流程，確保兩種模式使用相同或明確對應的解鎖機制

### 3.4 音效 `setTimeout` 改為排程器

- **現狀**：`soundManager` 中 `merge`、`build`、`startWave` 使用 `setTimeout` 串接多個音效，頻繁呼叫時會堆疊大量 timeout
- **解法**：改為音效序列管理器，以 `requestAnimationFrame` 或內部排程驅動音效播放
